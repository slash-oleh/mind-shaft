# General: Code review

## Define the PR review and merge responsibilities
Establish clear ownership for Pull Requests (PRs) and designate the author as the primary driver of the process.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/code-review/define-the-pr-review-and-merge-responsibilities.md)

## Don't harm
Leave the code better than you found it by fixing small issues encountered during development and ensuring new changes do not degrade the overall codebase health.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/code-review/don-t-harm.md)

## Don't keep working on merged branches
Stop pushing commits to a branch once it has been merged into the main development line. Create a new branch for any subsequent work or follow-up fixes.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/code-review/don-t-keep-working-on-merged-branches.md)

## Keep formatting and style rules in the linter
Automate code formatting and style enforcement using linters and formatters like ESLint or Prettier.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/code-review/keep-formatting-and-style-rules-in-the-linter.md)

## Keep PRs small
Keep Pull Requests (PRs) small and focused, ideally under 500 lines of code changes.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/code-review/keep-prs-small.md)

## Perform code review
Conduct code reviews promptly and professionally, focusing on defect detection and architectural alignment.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/code-review/perform-code-review.md)

## Prioritize architectural and design feedback in reviews
Prioritize high-level concerns such as architecture, design patterns, and overall logic during code reviews before addressing minor implementation details.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/code-review/prioritize-architectural-and-design-feedback-in-reviews.md)

## Reference project guidelines in reviews
Reference established project guidelines and documentation in code review comments instead of writing repetitive explanations.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/code-review/reference-project-guidelines-in-reviews.md)

## Reference similar issues in comments
Provide a detailed explanation for the first occurrence of an issue within a Pull Request and link to it for all subsequent occurrences instead of duplicating the feedback.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/code-review/reference-similar-issues-in-comments.md)
