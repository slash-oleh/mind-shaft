# General: Generic

## Apply clean architecture approaches
Separate business logic from infrastructure concerns like HTTP handling, database queries, and external APIs.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/generic/apply-clean-architecture-approaches.md)

## Automate build, test and deployment
Automate build, test, and deployment processes to ensure consistency and reliability across environments.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/generic/automate-build-test-and-deployment.md)

## Avoid premature optimization
Prioritize code readability and maintainability over performance until a specific performance issue is identified and measured.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/generic/avoid-premature-optimization.md)

## Change style for single item case
Use the simplest possible syntax for simple cases and refactor to complex structures only when necessary.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/generic/change-style-for-single-item-case.md)

## Don't use outdated tools
Use tools and libraries with active maintenance and healthy ecosystems to ensure long-term project stability.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/generic/don-t-use-outdated-tools.md)

## Keep files size reasonable
Maintain a balanced file size to ensure code remains navigable and focused without becoming fragmented.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/generic/keep-files-size-reasonable.md)

## Keep README up-to-date
Maintain the README as a fresh, accurate, and comprehensive entry point for the project.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/generic/keep-readme-up-to-date.md)

## Prefer composition over inheritance
Use composition to build complex functionality by combining simple, independent components instead of creating deep class hierarchies.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/generic/prefer-composition-over-inheritance.md)

## Resolve linter issues before committing to the mainline
Resolve all linter errors and warnings before merging code into the main branch.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/generic/resolve-linter-issues-before-committing-to-the-mainline.md)

## Use Datetime ISO format
Use the [ISO 8601](https://en.wikipedia.org/wiki/ISO_8601) format for Date, Time, and Duration representation.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/generic/use-datetime-iso-format.md)

## Use guard checks
Apply guard checks at the beginning of functions to handle edge cases early and use robust range comparisons instead of strict equality for threshold checks.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/generic/use-guard-checks.md)

## Use text checkers
Utilize automated text, grammar, and style checkers for all written documentation and code comments.
[read more](https://github.com/insolite/dev-rules/blob/main/src/general/generic/use-text-checkers.md)
