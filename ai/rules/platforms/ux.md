# Platforms: UX

## Add explanation to disabled controls
Provide a visible explanation of *why* a control is disabled and *what* the user can do to enable it.
[read more](https://github.com/insolite/dev-rules/blob/main/src/platforms/ux/add-explanation-to-disabled-controls.md)

## Avoid floating areas
Minimize the use of modal windows, pop‑overs, and other floating UI elements.
[read more](https://github.com/insolite/dev-rules/blob/main/src/platforms/ux/avoid-floating-areas.md)

## Don't implement unjustifiably complex design traits without clarifying
Clarify unusually complex or inconsistent design details with the designer before implementation.
[read more](https://github.com/insolite/dev-rules/blob/main/src/platforms/ux/don-t-implement-unjustifiably-complex-design-traits-without-clarifying.md)

## Don't wrap interactive elements into each other
Avoid nesting interactive elements (buttons, links, inputs) inside other interactive elements.
[read more](https://github.com/insolite/dev-rules/blob/main/src/platforms/ux/don-t-wrap-interactive-elements-into-each-other.md)

## Limit component variations
Standardize on a small set of well-defined component variants. Avoid creating multiple components or variations that serve the same underlying purpose.
[read more](https://github.com/insolite/dev-rules/blob/main/src/platforms/ux/limit-component-variations.md)

## Limit simultaneous components
Limit the number of interactive elements, data points, and navigation options displayed simultaneously.
[read more](https://github.com/insolite/dev-rules/blob/main/src/platforms/ux/limit-simultaneous-components.md)

## Limit the use of color
Use color sparingly and purposefully. A restrained color palette improves clarity, focuses attention, and makes meaningful color cues (like errors or success states) stand out more effectively.
[read more](https://github.com/insolite/dev-rules/blob/main/src/platforms/ux/limit-the-use-of-color.md)

## Maintain core accessibility features
Preserve and enhance native browser accessibility features.
[read more](https://github.com/insolite/dev-rules/blob/main/src/platforms/ux/maintain-core-accessibility-features.md)

## Make form fail early
Validate form inputs as soon as the user interacts with them instead of waiting for submission.
[read more](https://github.com/insolite/dev-rules/blob/main/src/platforms/ux/make-form-fail-early.md)

## Prefer fail-tolerant approaches
Design interfaces that prevent errors before they happen and allow users to recover easily.
[read more](https://github.com/insolite/dev-rules/blob/main/src/platforms/ux/prefer-fail-tolerant-approaches.md)

## Prefer flat menus over nested submenus
Keep navigation shallow by using flat lists with clear grouping instead of nested submenus.
[read more](https://github.com/insolite/dev-rules/blob/main/src/platforms/ux/prefer-flat-menus-over-nested-submenus.md)

## Use high-visibility controls for few options
Use button groups, switches, or radio buttons instead of dropdowns for small sets of options (typically 2-5).
[read more](https://github.com/insolite/dev-rules/blob/main/src/platforms/ux/use-high-visibility-controls-for-few-options.md)

## Use visual accents
Establish a clear visual hierarchy using variations in font weight, color, size, and positioning to guide user attention.
[read more](https://github.com/insolite/dev-rules/blob/main/src/platforms/ux/use-visual-accents.md)
