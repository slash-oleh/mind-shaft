# Rules

Best practices and contribution guidelines for software developers.

## Motivation

We describe best practices and decision/solution recommendations for common
software development problems. Apart from
[design patterns](https://en.wikipedia.org/wiki/Software_design_pattern) it
describes more simple and practical cases and provides a centralized
comprehensive list of things software developer must know before contributing to
a project.

Every recommendation article is defined in vacuum. You should consider project
technical limitations and effort in a particular situation to decide whether to
follow them or not.

## Impact

We define the argumentation through so-called **Impact**. They are
referencing the well-known positive or negative consequences for the product.

All characteristics (the "why" part) are described through the **Impact**. Impact
items are argued separately from recommendation items. Every impact item can
imply either positive or negative consequences.

The articles should avoid using ambiguous characteristics such as words:
_better_, _improved_, _increased_, _flexible_, _obvious_, _explicit_, _easy_, etc.
without definition or arguing.

We also don't use these as Impact:

- Speed (Velocity)
- Quality
- Cost (Price)
- Tech Debt - goes automatically with any bad solution we decide to keep

Impact items should be more concrete, rather be **ways** to achieve these as
goals.
