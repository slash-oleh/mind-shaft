# Keep files size reasonable

Maintain a balanced file size to ensure code remains navigable and focused without becoming fragmented.

## Problem

Very large files (e.g., 1000+ lines) often become "god objects" or unorganized dumping grounds for unrelated logic, making them extremely difficult to read, test, and maintain. On the other hand, over-splitting code into a massive number of tiny files can lead to fragmentation, making it hard to follow the logic flow and increasing the mental overhead of navigating the project structure.

## Good solution

Aim for a reasonable size (e.g., up to 500 lines) as a general guideline. Group related logic, components, or services together to maintain high cohesion.

**Avoid over-fragmentation**: If a piece of code is small (only a few lines) and only used in one place, keep it inlined or in the same file as its usage to preserve context. Do not create a separate file for a tiny part unless it is consumed by multiple unrelated modules.

```ts
// Good: A focused service with related operations
// src/services/user-service.ts (~200 lines)
export const DEFAULT_USER_AVATAR = "https://example.com/avatar.png";

export const getUserProfile = (id: string, avatar = DEFAULT_USER_AVATAR) => { ... };
export const updateUserPreferences = (data: Prefs) => { ... };
```

## Bad solution

Creating monolithic files that handle multiple responsibilities or splitting every single utility function or constant into its own separate file regardless of its context or size.

```ts
// Bad: A massive file mixing unrelated concerns
// src/utils/everything.ts (2500 lines)
// contains: formatting, API calls, UI constants, math helpers...

// Bad: Over-fragmentation
// Extracting a single-use constant to a separate file instead of keeping it in context.
// src/services/user-service/constants.ts
export const DEFAULT_USER_AVATAR = "https://example.com/avatar.png";

// src/services/user-service.ts
import { DEFAULT_USER_AVATAR } from './constants';
```

## Impact

- **[Readability](../../home/impact/positive/readability.md)**: Focused files are easier to scan, and inlining small logic keeps the context available where it's needed.
- **[Maintainability](../../home/impact/positive/maintainability.md)**: Smaller files reduce merge conflicts, while avoiding fragmentation reduces the cost of navigating the project.
- **[Cohesion](../../home/impact/positive/cohesion.md)**: Keeping related logic together makes the system's architecture more intuitive.
- **[Coupling](../../home/impact/negative/coupling.md)**: Smaller files reduce the number of dependencies between modules.

## Exceptions

- When a file represents a logically atomic but complex entity that would become less clear if split.
- Autogenerated files (e.g., database schemas, API clients) and large configuration files.
