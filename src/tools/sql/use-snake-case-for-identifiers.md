# Use snake case for identifiers

Use `snake_case` (all lowercase words separated by underscores) for all database identifiers, including tables, columns, constraints, sequences, and indexes.

## Problem

Many relational databases (like PostgreSQL) automatically fold unquoted identifiers to lowercase or uppercase. If you create a table or column using `camelCase` or `PascalCase`, you often have to wrap the identifier in double quotes (e.g., `"userId"`) in every single SQL query to prevent the database from looking for `userid` and failing. This litters queries with unnecessary quotation marks, makes raw SQL painful to write, and frequently causes bugs when a query generator forgets the quotes.

## Good solution

Use complete `snake_case` for all database-level names. They are inherently case-insensitive, meaning they work perfectly without quotes in any SQL dialect or tool.

```sql
-- Good: Works universally without quotes in raw SQL
CREATE TABLE user_account (
    id UUID PRIMARY KEY,
    first_name VARCHAR(100),
    created_at TIMESTAMP
);

ALTER TABLE user_account ADD CONSTRAINT fk_user_account_role FOREIGN KEY ...
```

## Bad solution

Using `camelCase` or `PascalCase`, which forces the use of double-quotes everywhere just to make queries function.

```sql
-- Bad: Requires quotes for every future statement
CREATE TABLE "UserAccount" (
    id UUID PRIMARY KEY,
    "firstName" VARCHAR(100),
    "createdAt" TIMESTAMP
);

SELECT "firstName" FROM "UserAccount"; -- Fragile and tedious
```

## Why

- **[Maintainability](../../home/quality-attributes/positive/maintainability.md)**: Writing raw SQL statements and migrations is significantly easier and less prone to casing-related syntax errors.
- **[Consistency](../../home/quality-attributes/positive/consistency.md)**: Aligns with universal relational database naming standards.
- **[Portability](../../home/quality-attributes/positive/portability.md)**: Makes your SQL schemas and scripts easier to run across different database dialects and management tools without fighting case-folding rules.

## Exceptions

- **Legacy Databases**: When extending an existing project or database that already established a different convention (like `PascalCase`), where internal consistency is more important than switching mid-project.
- **Dependency-Managed Tables**: Tables or columns generated by third-party libraries (e.g., some audit loggers or auth providers) that you do not have direct control over.
